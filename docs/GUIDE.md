## 1. Что это такое

Pora AI — stateless HTTP-микросервис на FastAPI, который Go-бэкенд приложения
Pora вызывает по REST. Сервис не хранит никаких данных (все данные живут в Go
и передаются в каждом запросе) и возвращает: прогноз «когда закончится
продукт», категорию товара (раздел магазина), лучший час для push-уведомления,
рекомендации рецептов, распарсенные по URL рецепты, кулинарные советы,
заскоупленный чат и гибридные подсказки для корзины. Один и тот же код
работает и с локальной Ollama (бесплатно), и с любым OpenAI-совместимым
облаком — различие только в переменных окружения.

```
Flutter ─REST→ Go backend ─REST→ Pora AI ──→ LLM (Ollama ИЛИ OpenAI-совместимый API)
```

Ключевое свойство — **graceful degrade**: если LLM-ключ не задан, сервис не
падает, а продолжает работать на детерминированной ML/стат-логике (подробнее
в [разделе 4](#4-запуск-вообще-без-llm-деградированный-режим)).

## 2. Бесплатный запуск: Docker + Ollama (рекомендуемый путь)

Полностью бесплатный вариант: LLM (модель qwen3) крутится локально в Ollama,
никаких API-ключей и подписок не нужно.

### 2.1. Требования

- **Docker Desktop** (macOS / Windows) или Docker Engine + Compose plugin
  (Linux). Проверка: `docker compose version` должна напечатать версию.
- **Свободный диск**: модель qwen3 занимает порядка 5 ГБ; с учётом образов
  Docker закладывайте ~8 ГБ.
- **Память**: для комфортной работы локальной модели желательно 8+ ГБ RAM,
  выделенных Docker.
- Свободные порты **8000** (сам сервис) и **11434** (Ollama).

### 2.2. Шаг 1 — поднять контейнеры

Из корня репозитория:

```bash
docker compose up -d
```

Ожидаемый результат — оба контейнера созданы и запущены:

```
[+] Running 3/3
 ✔ Volume "pora-ai_ollama_models"  Created
 ✔ Container pora-ollama           Started
 ✔ Container pora-ai               Started
```

Проверить состояние можно так:

```bash
docker compose ps
```

Оба сервиса (`pora-ollama`, `pora-ai`) должны быть в статусе `Up` / `running`.

### 2.3. Шаг 2 — скачать модель (один раз)

```bash
docker compose exec ollama ollama pull qwen3
```

Ожидаемый результат — прогресс-бары слоёв и в конце:

```
pulling manifest
pulling a1b2c3...  100% ▕████████████████▏ 5.2 GB
...
verifying sha256 digest
writing manifest
success
```

- **Сколько ждать**: зависит от канала; ~5 ГБ при 100 Мбит/с — около 7–10
  минут, при медленном интернете — заметно дольше. Прерванную загрузку можно
  просто перезапустить той же командой — докачает.
- **Где хранится**: в именованном Docker-томе `ollama_models`
  (смонтирован в контейнер как `/root/.ollama`). Модель переживает
  перезапуски и пересборки контейнеров — скачивать её нужно только один раз.
- Проверить, что модель на месте:

```bash
docker compose exec ollama ollama list
```

```
NAME            ID              SIZE      MODIFIED
qwen3:latest    ...             5.2 GB    ...
```

### 2.4. Шаг 3 — проверить, что сервис жив

```bash
curl http://localhost:8000/health
```

Ожидаемый ответ:

```json
{
  "status": "ok",
  "version": "2.0.0",
  "llm_enabled": true,
  "models": {"main": "qwen3", "fast": "qwen3"},
  "sections": ["dairy", "produce", "bakery", "pantry", "drinks", "meat_fish", "other"],
  "fast_langs": ["en", "ru"],
  "refusal_langs": ["ar", "de", "en", "es", "fr", "he", "hi", "it", "ja", "ko", "pl", "pt", "ru", "tr", "zh"]
}
```

Важно: `"llm_enabled": true` означает лишь, что ключ (`LLM_API_KEY=ollama`,
любой непустой) задан. Если модель ещё не скачана, LLM-вызовы будут падать и
сервис молча откатится на фолбэки — сначала выполните шаг 2.

### 2.5. Шаг 4 — первый осмысленный запрос

```bash
curl -X POST http://localhost:8000/v1/categorize \
  -H 'Content-Type: application/json' \
  -d '{"names": ["авокадо", "кефир"], "lang": "ru"}'
```

Ожидаемый ответ (быстрый классификатор, LLM даже не вызывается):

```json
{"results": [
  {"name": "авокадо", "section": "produce", "section_label": "Овощи и фрукты",
   "confidence": 0.75, "lang": "ru", "method": "fast"},
  {"name": "кефир", "section": "dairy", "section_label": "Молочное",
   "confidence": 0.8, "lang": "ru", "method": "fast"}
]}
```

(Числа `confidence` у вас будут немного другими — это вероятности модели.)

Интерактивная документация (Swagger UI), где можно тыкать все эндпоинты из
браузера: **http://localhost:8000/docs**

### 2.6. Остановить / перезапустить / удалить

| Действие | Команда | Что происходит |
|---|---|---|
| Остановить | `docker compose stop` | Контейнеры остановлены, данные и модель на месте |
| Запустить снова | `docker compose start` (или `up -d`) | Продолжает с того же места |
| Перезапустить один сервис | `docker compose restart pora-ai` | Например, после смены env |
| Удалить контейнеры | `docker compose down` | Том `ollama_models` (модель) **сохраняется** |
| Удалить всё, включая модель | `docker compose down -v` | Стирает и том — модель придётся качать заново |
| Пересобрать после правок кода | `docker compose up -d --build` | Пересборка образа pora-ai |

### 2.7. Вариант легче по ресурсам — маленькая модель

Если машина слабая (мало RAM, нет GPU), возьмите компактную модель, например
`qwen3:4b` — она в разы меньше и быстрее на CPU (качество чуть ниже):

1. В `docker-compose.yml` в блоке `environment` сервиса `pora-ai` замените:

   ```yaml
   LLM_MODEL: "qwen3:4b"
   ```

2. Примените и скачайте модель:

   ```bash
   docker compose up -d
   docker compose exec ollama ollama pull qwen3:4b
   ```

Никаких изменений в коде не нужно — модель выбирается только через env
`LLM_MODEL`.

Для GPU-ускорения раскомментируйте блок `deploy.resources` у сервиса
`ollama` в `docker-compose.yml` (комментарий уже есть в файле).

## 3. Запуск с облачным LLM (опционально)

Сервис работает с **любым OpenAI-совместимым API** — меняются только три
переменные окружения, код один и тот же.

### 3.1. Пример: OpenAI

В `docker-compose.yml` замените блок переменных сервиса `pora-ai`:

```yaml
environment:
  LLM_BASE_URL: "https://api.openai.com/v1"
  LLM_API_KEY:  "sk-..."          # ваш ключ
  LLM_MODEL:    "gpt-4o-mini"
```

и перезапустите:

```bash
docker compose up -d
```

Контейнер `ollama` при этом больше не нужен — можно оставить (не мешает) или
запускать только сервис: `docker compose up -d pora-ai`.

### 3.2. Бесплатные / дешёвые альтернативы

Любой провайдер с OpenAI-совместимым endpoint подходит — просто подставьте
его `base_url`, ключ и имя модели:

| Провайдер | `LLM_BASE_URL` | Примечание |
|---|---|---|
| **Groq** | `https://api.groq.com/openai/v1` | Есть бесплатный tier, очень быстрый inference (актуальные имена моделей смотрите в их консоли) |
| **OpenRouter** | `https://openrouter.ai/api/v1` | Агрегатор; есть модели с пометкой `:free` |
| OpenAI | `https://api.openai.com/v1` | Платно; `gpt-4o-mini` — дешёвый вариант |
| Together / Fireworks / Mistral и др. | см. их доки | Всё, что говорит на OpenAI-протоколе |

Требование к провайдеру: поддержка **structured output**
(`response_format=json_schema`) — она используется категоризацией и парсингом
рецептов. Если провайдер её не поддерживает, эти вызовы вернут фолбэки
(`other` / пустой рецепт), сервис при этом не упадёт.

**Безопасность:** ключ живёт только на сервере (env контейнера). Никогда не
передавайте его во Flutter/браузер.

## 4. Запуск вообще без LLM (деградированный режим)

Тоже полностью бесплатно и мгновенно: оставьте `LLM_API_KEY` **пустым** — 
функция `llm_enabled()` вернёт `False`, и все LLM-пути аккуратно откатятся на
фолбэки. Сервис никогда не бросает ошибку из-за отсутствия ключа.

```bash
# без Docker:
LLM_API_KEY= uvicorn main:app --port 8000
# или в docker-compose.yml:  LLM_API_KEY: ""
```

Что **работает полностью** (чистая ML/стат-логика, LLM не нужен):

- `/v1/replenishment` — прогноз пополнения;
- `/v1/notify-time` — лучший час пуша;
- `/v1/recommend` — рекомендация рецепта из каталога;
- `/v1/suggest` — все типы подсказок, кроме `dish`;
- `/v1/categorize` — быстрый классификатор для **ru/en**;
- `/v1/parse-recipe` — путь через JSON-LD (если разметка есть на странице);
- `/v1/brief` — работает, но `tip` внутри будет фолбэком.

Что **деградирует**:

- `/v1/chat` — вместо ответа возвращает локализованный отказ с пометкой
  `"note": "llm_disabled"` (HTTP 200, не ошибка);
- `/v1/tip` — отдаёт заготовленный совет, `"source": "fallback"`;
- `/v1/categorize` для языков, отличных от ru/en, и для кастомных `sections`
  — вернёт `"other"` (или первую кастомную секцию) с `confidence: 0.0`;
- `/v1/parse-recipe` для страниц без JSON-LD — пустой список ингредиентов,
  `"source": "none"`;
- `/v1/suggest` — не будет подсказок типа `dish`.

## 5. Локальный запуск без Docker

Нужен Python 3.12 (см. `Dockerfile`; более свежие 3.x, как правило, тоже
подходят).

### 5.1. Установка и запуск

```bash
pip install -r requirements.txt
uvicorn main:app --port 8000 --reload
```

Ожидаемый вывод:

```
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Application startup complete.
```

Без env-переменных это деградированный режим ([раздел 4](#4-запуск-вообще-без-llm-деградированный-режим)):
`LLM_API_KEY` пуст, LLM выключен.

### 5.2. С Ollama на хосте

Если Ollama установлена прямо на машине (не в Docker,
[ollama.com](https://ollama.com)):

```bash
ollama pull qwen3          # один раз
LLM_BASE_URL=http://localhost:11434/v1 LLM_API_KEY=ollama LLM_MODEL=qwen3 \
  uvicorn main:app --port 8000 --reload
```

Замечание: `http://localhost:11434/v1` и `qwen3` — это и есть значения по
умолчанию в коде (`pora_llm.py`), так что достаточно задать только
`LLM_API_KEY=ollama` (любой непустой). Но явно прописать все три — надёжнее.

### 5.3. С облачным ключом

```bash
LLM_BASE_URL=https://api.openai.com/v1 LLM_API_KEY=sk-... LLM_MODEL=gpt-4o-mini \
  uvicorn main:app --port 8000 --reload
```

## 6. Полный справочник эндпоинтов

Общие правила:

- Все запросы и ответы — `application/json`, UTF-8.
- Поле `lang` везде **опционально**: не передали — сервис определит язык сам
  по тексту (`detect_lang`, по скрипту/диакритике).
- Невалидное тело запроса (не тот тип, нет обязательного поля) → **422**
  (стандартная валидация FastAPI/Pydantic).
- Кривая дата в `today`/`purchases` → **400** `{"detail": "bad date: ..."}`.
- Ошибка внешнего мира (сайт не отдался в `/v1/parse-recipe`) → **502**.
- Локализованные отказы чата — это **HTTP 200** с `refused: true`, не ошибка.

Краткая карта:

| Метод и путь | Назначение | Нужен ли LLM |
|---|---|---|
| `GET /health` | Проверка живости и конфигурации | нет |
| `GET /metrics` | Телеметрия: LLM-вызовы/ошибки/латенси/токены + статистика кэшей | нет |
| `POST /v1/replenishment` | Прогноз «когда закончится продукт» | нет |
| `POST /v1/categorize` | Раздел магазина для названий товаров | нет для ru/en; да для остальных языков и кастомных секций |
| `POST /v1/notify-time` | Лучший час push-уведомления | нет |
| `POST /v1/recommend` | Один рецепт под вкус пользователя | нет |
| `POST /v1/parse-recipe` | Извлечь рецепт по URL | нет (JSON-LD) / да (фолбэк) |
| `POST /v1/tip` | Короткий кулинарный совет | да (без LLM — фолбэк) |
| `POST /v1/chat` | Заскоупленный чат про еду и покупки | да (без LLM — отказ) |
| `POST /v1/brief` | Дневной брифинг одним вызовом | частично |
| `POST /v1/suggest` | Гибридные подсказки (4 типа) | нет, кроме типа `dish` |

### 6.1. `GET /health`

Проверка живости. Тела нет.

```bash
curl http://localhost:8000/health
```

Ответ — см. [шаг 2.4](#24-шаг-3--проверить-что-сервис-жив). Поля: `status`,
`version`, `llm_enabled` (задан ли ключ), `models` (какая модель используется
как основная `main` и какая как быстрая `fast` — совпадают, если
`LLM_MODEL_FAST` не задана), `sections` (канонические ключи разделов),
`fast_langs` (языки быстрого классификатора), `refusal_langs` (языки
локализованных отказов).

### 6.1.1. `GET /metrics` — телеметрия

Кумулятивные счётчики с момента старта процесса. Тела нет.

```bash
curl http://localhost:8000/metrics
```

```json
{
  "llm": {
    "uptime_s": 3600.5,
    "llm_calls": {"fast": 120, "main": 15},
    "llm_errors": {"fast": 2, "main": 0},
    "llm_latency_ms_total": {"fast": 84000.0, "main": 30500.0},
    "tokens": {"prompt": 45000, "completion": 9000}
  },
  "caches": {
    "categorize": {"size": 340, "hits": 812, "misses": 410},
    "recipe": {"size": 12, "hits": 40, "misses": 14}
  }
}
```

Среднюю латенсию считайте на стороне потребителя:
`llm_latency_ms_total[kind] / llm_calls[kind]`. Токены доступны, только если
провайдер возвращает `usage` (облачные — да, Ollama — зависит от версии).

### 6.2. `POST /v1/replenishment` — прогноз пополнения

Считает по истории покупок, когда продукт закончится: медианный интервал
между покупками, уверенность (1 − коэффициент вариации), статус-бакет.

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `today` | `string` (ISO-дата) | нет (по умолчанию — сегодня) | Точка отсчёта |
| `purchases` | `[{product: string, date: string}]` | **да** | История покупок; `date` — ISO-дата |

**Правило:** на продукт нужно **≥ 3** покупок, иначе он пропускается.

```bash
curl -X POST http://localhost:8000/v1/replenishment \
  -H 'Content-Type: application/json' \
  -d '{
    "today": "2026-07-08",
    "purchases": [
      {"product": "Молоко", "date": "2026-06-24"},
      {"product": "Молоко", "date": "2026-07-01"},
      {"product": "Молоко", "date": "2026-07-07"}
    ]
  }'
```

**Ответ:**

```json
{
  "today": "2026-07-08",
  "predictions": [
    {"product": "Молоко", "every_days": 6.5, "due_date": "2026-07-13",
     "days_left": 5, "confidence": 0.92, "status": "ok", "events": 3}
  ]
}
```

Статусы: `overdue` (просрочено, `days_left < 0`), `due` (≤ 1 дня), `soon`
(≤ 3 дней), `ok` (дальше). Список отсортирован по `days_left` по возрастанию.

**Ошибки:** 400 при невалидной дате, 422 при невалидном теле.

### 6.3. `POST /v1/categorize` — раздел магазина

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `names` | `list[string]` | **да** | Названия товаров |
| `lang` | `string` | нет | `ru`/`en`/… ; не задан — автоопределение по каждому имени |
| `sections` | `list[string]` | нет | Своя таксономия разделов (ключи). Не задана — используются дефолтные `dairy, produce, bakery, pantry, drinks, meat_fish, other` |
| `section_labels` | `dict[string, string]` | нет | Локализованные подписи для кастомных секций; без него подписью будет сам ключ |

**Роутинг** (живёт в `main.py`):

- `sections` не задано и `lang ∈ {ru, en}` → быстрый локальный классификатор
  (TF-IDF + LogisticRegression); если его `confidence < 0.45` **и** LLM
  включён — эскалация в LLM;
- `sections` не задано и язык другой → LLM по каждому имени;
- `sections` задано → **один батчевый** LLM-вызов на весь список
  (независимо от языка).

Пример с дефолтной таксономией:

```bash
curl -X POST http://localhost:8000/v1/categorize \
  -H 'Content-Type: application/json' \
  -d '{"names": ["сметана", "бородинский хлеб", "форель"], "lang": "ru"}'
```

```json
{"results": [
  {"name": "сметана", "section": "dairy", "section_label": "Молочное",
   "confidence": 0.81, "lang": "ru", "method": "fast"},
  {"name": "бородинский хлеб", "section": "bakery", "section_label": "Хлеб и выпечка",
   "confidence": 0.77, "lang": "ru", "method": "fast"},
  {"name": "форель", "section": "meat_fish", "section_label": "Мясо и рыба",
   "confidence": 0.69, "lang": "ru", "method": "fast"}
]}
```

Пример с **кастомными секциями** (нужен работающий LLM):

```bash
curl -X POST http://localhost:8000/v1/categorize \
  -H 'Content-Type: application/json' \
  -d '{
    "names": ["стейк", "шпинат", "молоко"],
    "sections": ["meat_section", "veggies", "dairy_section", "other"],
    "section_labels": {
      "meat_section":  "Мясной отдел",
      "veggies":       "Овощи и зелень",
      "dairy_section": "Молочный отдел",
      "other":         "Прочее"
    },
    "lang": "ru"
  }'
```

```json
{"results": [
  {"name": "стейк", "section": "meat_section", "section_label": "Мясной отдел",
   "confidence": 0.9, "lang": "ru", "method": "llm"},
  {"name": "шпинат", "section": "veggies", "section_label": "Овощи и зелень",
   "confidence": 0.9, "lang": "ru", "method": "llm"},
  {"name": "молоко", "section": "dairy_section", "section_label": "Молочный отдел",
   "confidence": 0.9, "lang": "ru", "method": "llm"}
]}
```

Если LLM выключен, при кастомных секциях каждое имя получит фолбэк-секцию
(`other`, если она есть в вашем списке, иначе первую) с `confidence: 0.0`.

Поля ответа: `name`, `section` (ключ), `section_label` (локализованная
подпись), `confidence` (0…1), `lang`, `method` (`fast` | `llm`).

**Ошибки:** 422 при невалидном теле.

### 6.4. `POST /v1/notify-time` — лучший час уведомления

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `opens` | `list[string]` (ISO datetime) | **да** | Моменты открытия приложения |

Невалидные таймстемпы молча пропускаются. Пик ищется с приоритетом вечернего
окна 16:00–21:59; при пустом входе — дефолт `hour: 18`.

```bash
curl -X POST http://localhost:8000/v1/notify-time \
  -H 'Content-Type: application/json' \
  -d '{"opens": ["2026-07-06T18:30:00", "2026-07-07T19:05:00", "2026-07-07T18:45:00"]}'
```

```json
{"hour": 18, "window_share": 1.0, "samples": 3}
```

`window_share` — доля всех заходов в окне ±1 час вокруг пика.

### 6.5. `POST /v1/recommend` — один рецепт под вкус

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `recipe_imports` | `list[string]` | нет (по умолчанию `[]`) | Какие рецепты пользователь уже импортировал/готовил |
| `regular_products` | `list[string]` | нет (`[]`) | Регулярные покупки |
| `lang` | `string` | нет | Не влияет на подбор, зарезервировано |
| `catalog` | `list[object]` | нет | Свой каталог рецептов: `[{"name", "cuisine", "ingredients": [...]}]`. Без него используется встроенный демо-каталог из 5 рецептов |

Определяет любимую кухню по импортам и подбирает непробованный рецепт с
максимальным пересечением ингредиентов с регулярными покупками. Каталог
можно передавать в каждом запросе (поле `catalog`) — Go-бэкенд может
хранить свой список блюд и прокидывать его сюда; встроенный каталог — лишь
фолбэк для демо. То же поле принимает `/v1/suggest`.

```bash
curl -X POST http://localhost:8000/v1/recommend \
  -H 'Content-Type: application/json' \
  -d '{"recipe_imports": ["Карбонара"], "regular_products": ["паста", "сыр", "молоко"]}'
```

```json
{"top_cuisine": "Итальянская", "recipe": "Мак-н-чиз",
 "cuisine": "Итальянская", "pantry_match": 0.75}
```

### 6.6. `POST /v1/parse-recipe` — извлечь рецепт по URL

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `url` | `string` | **да** | URL страницы с рецептом |
| `lang` | `string` | нет | Влияет на заголовок `Accept-Language` при скачивании |
| `sections` | `list[string]` | нет | Кастомная таксономия для секций ингредиентов (как в categorize) |

Пайплайн: скачивание страницы (браузероподобные заголовки, ретраи на
429/5xx, обрезка до 400 КБ) → **бесплатный путь** JSON-LD
(`@type: "Recipe"`) → LLM-фолбэк по тексту страницы с
анти-галлюцинационной проверкой (ингредиенты, которых нет в исходном тексте,
выбрасываются) → каждому ингредиенту проставляется раздел магазина.

```bash
curl -X POST http://localhost:8000/v1/parse-recipe \
  -H 'Content-Type: application/json' \
  -d '{"url": "https://example.com/recepty/borshch", "lang": "ru"}'
```

**Ответ:**

```json
{
  "title": "Борщ классический",
  "ingredients": [
    {"raw": "Свёкла 2 шт", "name": null, "qty": null, "unit": null, "section": "produce"},
    {"raw": "Говядина 500 г", "name": null, "qty": null, "unit": null, "section": "meat_fish"},
    {"raw": "Капуста 300 г", "name": null, "qty": null, "unit": null, "section": "produce"}
  ],
  "source": "jsonld"
}
```

- `source` ∈ `jsonld` (нашлась разметка, LLM не тратился) | `llm`
  (извлечено моделью) | `none` (ничего не извлечено);
- на JSON-LD-пути `name/qty/unit` обычно `null` — там только сырые строки
  `raw`; LLM-путь заполняет их;
- без LLM-ключа работает только JSON-LD-путь.

**Ошибки:** **502** `{"detail": "fetch/parse failed: ..."}`, если страница
не скачалась или упал парсинг; 422 при невалидном теле.

### 6.7. `POST /v1/tip` — короткий совет

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `top_cuisine` | `string` | нет (по умолчанию `"Итальянская"`) | Любимая кухня |
| `frequent` | `list[string]` | нет (`[]`) | Частые покупки |

```bash
curl -X POST http://localhost:8000/v1/tip \
```

### 6.8. `POST /v1/chat` — заскоупленный чат

Отвечает **только** про еду, рецепты, покупки. Офтоп (код, юриспруденция,
медицина и т.п.) режется дешёвым regex-фильтром ещё **до** LLM-вызова —
токены не тратятся.

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `message` | `string` | **да** | Сообщение пользователя |
| `lang` | `string` | нет | Иначе автоопределение |

```bash
curl -X POST http://localhost:8000/v1/chat \
  -H 'Content-Type: application/json' \
  -d '{"message": "Что приготовить из кабачков на ужин?"}'
```

Ответ (on-topic, LLM включён):

```json
{"text": "Попробуйте оладьи из кабачков: натрите 2 кабачка, ...", "lang": "ru", "refused": false}
```

Офтоп (например, «напиши python код») — HTTP 200:

```json
{"text": "Я помогаю только с едой и покупками 🙂", "lang": "ru", "refused": true}
```

LLM выключен, но вопрос on-topic:

```json
{"text": "Я помогаю только с едой и покупками 🙂", "lang": "ru", "refused": false, "note": "llm_disabled"}
```

**Тело:**

### 6.10. `POST /v1/suggest` — гибридные подсказки

Главный «умный» эндпоинт: объединяет статистику покупок, историю рецептов,
регулярные товары и текущую корзину в один отсортированный список подсказок
четырёх типов.

**Тело:**

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `today` | `string` (ISO-дата) | нет | |
| `purchases` | `list[{product, date}]` | нет (`[]`) | Для типа `replenish` |
| `recipe_imports` | `list[string]` | нет (`[]`) | Импортированные рецепты (предпочтения по кухне) |
| `regular_products` | `list[string]` | нет (`[]`) | Регулярные покупки |
| `current_cart` | `list[string]` | нет (`[]`) | Что уже в корзине — для типа `basket_fit` |
| `lang` | `string` | нет (по умолчанию `"en"`) | Язык поля `reason` |
| `limit` | `int` | нет (по умолчанию `5`) | Диапазон **1…20**, иначе 422 |

**Четыре типа подсказок:**

| `type` | Смысл | Источник | Score |
|---|---|---|---|
| `basket_fit` | «Подходит к корзине!» — в корзине есть товар из каталожного рецепта, предлагается недостающий ингредиент | brain (локально) | `0.6 + 0.3·(товар в regular_products) + 0.2·(рецепт не пробован)` |
| `replenish` | «Скоро закончится — пора пополнить» — прогноз overdue/due/soon | brain | `urgency × confidence`, urgency: overdue 1.0, due 0.8, soon 0.6 |
| `recipe` | «Подойдёт под ваш вкус» — рецепт из каталога | brain | `pantry_match + 0.3·(совпала любимая кухня)` |
| `dish` | Блюдо, придуманное LLM под предпочтения | LLM | фиксированно `0.5` |

Внутренние лимиты до слияния: до 3 `replenish`, до 3 `basket_fit`,
до 2 `recipe`, максимум 1 `dish`. Затем всё сливается, дубликаты по
`(type, product, recipe)` схлопываются, сортировка по `score` убыв.,
обрезка до `limit`.

```bash
curl -X POST http://localhost:8000/v1/suggest \
  -H 'Content-Type: application/json' \
  -d '{
    "today": "2026-07-08",
    "purchases": [
      {"product": "Молоко", "date": "2026-06-24"},
      {"product": "Молоко", "date": "2026-07-01"},
      {"product": "Молоко", "date": "2026-07-07"}
    ],
    "recipe_imports": ["Карбонара"],
    "regular_products": ["пармезан", "молоко", "яйца"],
    "current_cart": ["спагетти", "бекон"],
    "lang": "ru",
    "limit": 5
  }'
```

**Ответ:**

**Ответ** (с этими данными «Молоко» ещё не скоро закончится — `replenish` не
попал в выдачу; в корзине спагетти и бекон — сработала «Карбонара»):

```json
{
  "lang": "ru",
  "today": "2026-07-08",
  "suggestions": [
    {"type": "basket_fit", "product": "пармезан", "recipe": "Карбонара",
     "reason": "Подходит к корзине!", "score": 0.9,
     "meta": {"matched_cart_item": "бекон", "cuisine": "Итальянская", "in_regular": true}},
    {"type": "basket_fit", "product": "яйца", "recipe": "Карбонара",
     "reason": "Подходит к корзине!", "score": 0.9,
     "meta": {"matched_cart_item": "бекон", "cuisine": "Итальянская", "in_regular": true}},
    {"type": "recipe", "product": null, "recipe": "Мак-н-чиз",
     "reason": "Подойдёт под ваш вкус", "score": 0.55,
     "meta": {"cuisine": "Итальянская", "pantry_match": 0.25, "missing": ["масло", "паста", "сыр"]}},
    {"type": "dish", "product": null, "recipe": "Лазанья болоньезе",
     "reason": "Попробуйте новое блюдо", "score": 0.5,
     "meta": {"top_cuisine": "Итальянская", "source": "llm"}},
    {"type": "recipe", "product": null, "recipe": "Лазанья",
     "reason": "Подойдёт под ваш вкус", "score": 0.3,
     "meta": {"cuisine": "Итальянская", "pantry_match": 0.0, "missing": ["паста", "помидоры", "сыр", "фарш"]}}
  ]
}
```

Логика «чего не будет»: без `current_cart` нет `basket_fit`; без `purchases`
нет `replenish`; без LLM-ключа нет `dish`.

**Ошибки:** 400 при кривой дате, 422 при `limit` вне 1…20 или невалидном
теле.

## 7. Env-переменные — полная таблица

| Переменная | Дефолт | Что делает |
|---|---|---|
| `LLM_BASE_URL` | `http://localhost:11434/v1` | Базовый URL OpenAI-совместимого API. Для Ollama в docker-compose — `http://ollama:11434/v1`; для OpenAI — `https://api.openai.com/v1` |
| `LLM_API_KEY` | `""` (пусто) | API-ключ. **Пусто → LLM выключен** (graceful degrade, см. раздел 4). Для Ollama годится любая непустая строка, например `ollama` |
| `LLM_MODEL` | `qwen3` | Имя модели у провайдера: `qwen3`, `qwen3:4b`, `gpt-4o-mini`, … |
| `LLM_MODEL_FAST` | *(не задана → = `LLM_MODEL`)* | Дешёвая модель для категоризации, советов и подсказок блюд. Основная модель (`LLM_MODEL`) остаётся для чата и парсинга рецептов. |
| `PORA_CACHE_ENABLED` | включено | In-process-кэш LLM-результатов (см. раздел 8). Значения `0`, `false`, `no` отключают; всё остальное (и отсутствие переменной) — включено. Читается **один раз при старте** процесса |

Все переменные читаются при импорте `pora_llm.py` — после изменения env
нужно перезапустить процесс/контейнер (`docker compose up -d` подхватит
изменения из `docker-compose.yml`).

## 8. Кэширование

Сервис держит два кэша **в памяти процесса** (LRU + TTL, поток-безопасные,
`_cache.py`), чтобы не звать LLM/сеть дважды за одно и то же:

| Кэш | Что кэширует | Объём | TTL |
|---|---|---|---|
| Категоризация | Результат LLM-категоризации по ключу «имя (lowercase) + набор секций» | 2048 записей | 1 час |
| Рецепты | Результат `parse-recipe` по ключу «URL + секции + lang» | 256 записей | 24 часа |

Детали поведения:

- кэшируются только **успешные** результаты (категоризация с высокой
  уверенностью; рецепты с `source` `jsonld`/`llm`) — неудачи не «залипают»;
- батчевая категоризация проверяет кэш **по каждому имени** и шлёт в LLM
  только промахи;
- кэш живёт в памяти — перезапуск контейнера его очищает; это нормально;
- быстрый ru/en-классификатор не кэшируется — он и так локальный и дешёвый.

**Зачем**: одни и те же товары («молоко», «хлеб») категоризуются у тысяч
пользователей, а популярные URL рецептов импортируются многократно. Кэш
превращает повторные обращения в бесплатные — это прямая экономия токенов
(денег в облаке) и времени (секунды инференса локальной модели).

**Как отключить** (например, при отладке промптов, чтобы видеть каждый
LLM-вызов):

```yaml
# docker-compose.yml → services → pora-ai → environment
PORA_CACHE_ENABLED: "0"
```

или локально: `PORA_CACHE_ENABLED=0 uvicorn main:app --port 8000`.

Размеры и TTL заданы константами в `constants.py`
(`CATEGORIZE_CACHE_SIZE/TTL_S`, `RECIPE_CACHE_SIZE/TTL_S`) — через env они
не меняются.

## 9. Тестирование

Для тестов **не нужны** ни Ollama, ни API-ключ: LLM в тестах всегда мокается
(fixture `mock_chat` в `tests/conftest.py` подменяет `pora_llm._chat`),
реальных сетевых вызовов нет.

```bash
pytest                         # весь suite (~секунды)
pytest tests/test_brain.py     # один файл
pytest -k suggest              # тесты, в имени которых есть "suggest"
pytest -k test_health          # один конкретный тест по имени
pytest -x --tb=long            # стоп на первой ошибке, подробный трейс
```

Конфигурация — в `pytest.ini`: тесты лежат в `tests/`, по умолчанию тихий
вывод (`-q --tb=short`). Файлы: `test_brain.py` (чистая ML-логика),
`test_pora_llm.py` (LLM-модуль с моками), `test_endpoints.py` (все эндпоинты
через TestClient), `test_cache.py`, `test_constants.py`.

Быстрый офлайн-демо-прогон (тоже без LLM-сервера, печатает живые ответы
эндпоинтов через TestClient):

```bash
python smoke_test.py
```

Ожидаемый финал вывода: `✅ офлайн smoke test пройден`.

При добавлении нового эндпоинта расширяйте и `smoke_test.py`, и `tests/`.

## 10. Troubleshooting / FAQ

**1. Порт 8000 занят** (`bind: address already in use`).
Либо освободите порт (`lsof -i :8000` покажет, кто держит), либо поменяйте
маппинг в `docker-compose.yml`: `"8080:8000"` — сервис станет доступен на
`http://localhost:8080`. Для uvicorn: `--port 8080`. То же самое с портом
11434 (Ollama).

**2. Модель не скачивается / загрузка обрывается.**
Проверьте свободный диск (модель ~5 ГБ) и логи:
`docker compose logs -f ollama`. Команду
`docker compose exec ollama ollama pull qwen3` можно безопасно запускать
повторно — докачает с места обрыва. За корпоративным прокси может
понадобиться настройка прокси для Docker.

**3. `connection refused` к Ollama / LLM-ответы пустые.**
Частые причины: (а) контейнер `ollama` не поднят — `docker compose ps`;
(б) модель не скачана — `docker compose exec ollama ollama list`;
(в) неправильный `LLM_BASE_URL`: **изнутри** docker-compose это
`http://ollama:11434/v1` (имя сервиса), а `http://localhost:11434/v1`
работает только с хоста. Важно: сервис при ошибках LLM не падает, а тихо
отдаёт фолбэки — если везде `method: "llm"` не появляется и `source:
"fallback"`, проверяйте именно связку с Ollama.

**4. Ответы медленные (десятки секунд).**
Для локальной модели на CPU это **норма**: inference без GPU медленный.
Варианты: маленькая модель (`qwen3:4b`, см. раздел 2.7), GPU-блок в
`docker-compose.yml`, либо облачный провайдер (раздел 3). Учтите, что
детерминированные эндпоинты (`replenishment`, `notify-time`, `recommend`,
`suggest` без `dish`, ru/en `categorize`) быстрые всегда — медленным бывает
только LLM-путь.

**5. Как смотреть логи.**

```bash
docker compose logs -f pora-ai    # логи сервиса (запросы, ошибки)
docker compose logs -f ollama     # логи LLM (загрузка модели, inference)
```

Без Docker логи uvicorn идут прямо в терминал.

**6. `/v1/parse-recipe` возвращает 502.**
`{"detail": "fetch/parse failed: ..."}` означает, что страницу не удалось
скачать: сайт лежит, отвечает ошибкой либо блокирует ботов (сервис
представляется браузером и ретраит 429/5xx, но всесильным это его не
делает). Проверьте URL в браузере; попробуйте другой сайт с рецептом. Если
страница скачалась, но рецепт не найден — это **не** 502, а ответ 200 с
`"source": "none"` и пустым списком ингредиентов (на странице нет ни
JSON-LD, ни распознаваемого текста рецепта, или LLM выключен).

**7. Категоризация всё время возвращает `other`.**
Две типовые причины: (а) LLM выключен (`llm_enabled: false` в `/health`), а
язык не ru/en или заданы кастомные `sections` — фолбэк по контракту, при
этом `confidence: 0.0`; (б) LLM включён, но не отвечает (см. пункт 3).
Для ru/en-названий без кастомных секций работает локальный классификатор —
он LLM не требует.

**8. Как сменить модель.**
Поменяйте `LLM_MODEL` в `docker-compose.yml` (или в env запуска), для
Ollama — скачайте её (`docker compose exec ollama ollama pull <имя>`), затем
`docker compose up -d`. Код менять не нужно. Скачанные модели можно
посмотреть: `docker compose exec ollama ollama list`, удалить ненужную:
`docker compose exec ollama ollama rm <имя>`.

**9. Где интерактивная документация?**
Swagger UI: **http://localhost:8000/docs** — там видны все эндпоинты со
схемами, и любой запрос можно выполнить из браузера кнопкой «Try it out».
Альтернатива — ReDoc: `http://localhost:8000/redoc`.

**10. Чат отказывается отвечать на нормальный вопрос.**
Проверьте, нет ли в сообщении «стоп-слов» офтоп-фильтра (упоминания кода:
`python`, `sql`, ` ``` `, `import` и т.п., а также юридической/медицинской
лексики) — фильтр намеренно грубый, он экономит токены до LLM-вызова. Ответ
с `"note": "llm_disabled"` — это не отказ по теме, а выключенный LLM.

**11. `422 Unprocessable Entity`.**
Тело запроса не соответствует схеме: не хватает обязательного поля
(`names`, `message`, `url`, `opens`, `purchases`), неверный тип или `limit`
вне 1…20. Текст ошибки FastAPI указывает точное поле.

**12. Изменил env — ничего не поменялось.**
Env читается один раз при старте процесса. После правки
`docker-compose.yml` выполните `docker compose up -d` (пересоздаст контейнер
с новыми переменными); локально — перезапустите uvicorn.
